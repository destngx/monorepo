**Description**:
Verify the capability-based market provider system end-to-end: routing logic selects correct providers per capability+type, fallback chains work when primary fails, caching is unified at capability level, and Svelte dashboard seamlessly integrates with generic capability endpoints.

- **Routing Verification**: Confirm each capability correctly routes to its configured provider chain (IFC→Fmarket, Equity→Vnstock, etc.).
- **Fallback Testing**: Verify secondary provider is tried when primary fails.
- **Caching Verification**: Unified capability-level cache prevents redundant upstream calls.
- **Dashboard Integration**: Svelte dashboard calls generic capability endpoints; Investment Terminal renders data correctly.
- **End-to-End Flow**: User clicks ticker in dashboard → capability request sent → correct provider routed → cached response returned → UI renders.

**Acceptance Criteria**:

- ✅ Capability routing verified: each type (equity, ifc, USD_VND, etc.) routes to correct primary provider.
- ✅ Fallback behavior tested: primary provider disabled → secondary provider attempts → response successful.
- ✅ Cache hit ratio verified: first call ≥1200ms (upstream), second call ≤100ms (cached), TTL expiry triggers refetch.
- ✅ Svelte dashboard successfully calls `/api/market/ticker`, `/api/market/exchange-rate`, `/api/market/fund-nav-history` endpoints.
- ✅ Investment Terminal loads and renders fund data, rates, and historical data without errors.
- ✅ Error scenarios tested: timeout, unsupported capability, invalid parameters → graceful error responses.
- ✅ Load test: 10 concurrent requests to same capability → cache prevents thundering herd, latency ≤500ms.
- ✅ CORS/browser integration: No console errors; headers properly set.
- ✅ Tests passing: `bunx nx run wealth-management-engine:test` (all routing and fallback tests included).
- ✅ `bunx nx run wealth-management-engine:test-e2e` fails with non-zero status if any e2e assertion fails.

**Implementation Notes**:

- **Routing Verification Tests**:
  - Unit test per capability type verifying correct provider selection from routing config
  - Test table: capability + type → expected provider chain
  - Mock routing config to test different scenarios (provider priority swaps, fallback-only chains)
  - Verify service calls `GetTicker(symbol, tickerType)` on correct provider

- **Fallback Testing**:
  - Mock primary provider to return error
  - Verify service attempts secondary provider
  - Verify success if secondary succeeds
  - Test chain exhaustion: all providers fail → comprehensive error with tried list
- **Cache Verification**:
  - Verify capability-level cache keys: `market:{capability}:{type}:{params_hash}`
  - First call should hit upstream (no cache entry)
  - Second call with same params should hit cache (TTL still valid)
  - Wait TTL + 1s, third call should miss cache (expired) and hit upstream
  - Verify cache size doesn't grow unbounded (test cache eviction if applicable)
- **Capability Endpoint Testing**:
  - Test each endpoint with valid parameters:
    - `GET /api/market/ticker?symbol=FUND001&type=ifc` → IFC ticket via Fmarket
    - `GET /api/market/ticker?symbol=ACB&type=equity` → equity ticker via Vnstock
    - `GET /api/market/exchange-rate?from=USD&to=VND` → Fmarket rates
    - `GET /api/market/fund-nav-history?symbol=FUND001` → Fmarket NAV history
    - `GET /api/market/gold-price` → Fmarket gold prices
    - `GET /api/market/bank-rates` → Fmarket bank rates
  - Verify response schemas match expected domain models
  - Verify status codes: 200 on success, 400 on invalid params, 504 on timeout
- **Dashboard Integration**:
  - Ensure `apps/wealth-management-dashboard` (Svelte) updated to call generic capability endpoints (not action-based)
  - Update `src/shared/api/market.ts` client to use `/api/market/*` endpoints
  - Verify Vite dev server proxy correctly routes to Go backend
  - Test Investment Terminal loads and renders:
    - Fund list (via `GetFundNavHistory`)
    - Exchange rate display (via `GetExchangeRate`)
    - Bank rates table (via `GetBankInterestRate`)
    - Gold prices (via `GetGoldPrice`)
    - Ticker details (via `GetTicker` with appropriate type)
- **End-to-End Test Sequence**:
  1. Start Go backend: `bunx nx serve wealth-management-engine`
  2. Start Svelte dashboard: `bunx nx serve wealth-management-dashboard`
  3. Navigate to Investment Terminal in dashboard
  4. Trigger each capability type (IFC ticker, equity ticker, exchange rate, etc.)
  5. Verify responses render correctly in UI
  6. Check browser dev tools:
     - First call: latency ~1200ms (upstream)
     - Second call same params: <100ms (cached)
     - After TTL expiry: ~1200ms again (refetch)
  7. Verify no CORS errors, network errors, or console warnings
- **Load Test Scenario**:
  - Simulate 10 concurrent requests to same capability (e.g., same IFC ticker)
  - Monitor cache hit rate:
    - First request: cache miss, upstream call
    - Requests 2-10: cache hits (all should use cached response)
    - Expected hit rate: ~90% (9 hits, 1 miss)
  - Verify max latency remains ≤ 500ms (even first uncached call shouldn't exceed)
  - Verify Upstash connection stays stable, no rate limit exceeded
- **Fallback Scenario Test** (optional, for robustness):
  - Temporarily disable Fmarket in routing config (e.g., change routing to use Vnstock only for IFC)
  - Verify system falls back correctly
  - Re-enable Fmarket, verify primary provider is used again
  - This tests config-driven routing without code changes
- **Error Scenarios Tested**:
  - **Timeout**: Primary provider exceeds 10s timeout → fallback tried → success if secondary responds
  - **Unsupported capability**: E.g., `GetTicker(symbol, "crypto")` not in routing config → 400 with clear error
  - **Invalid parameters**: E.g., empty symbol → 400 with validation error details
  - **All providers fail**: All providers in chain fail → comprehensive error listing tried providers
  - **Empty results**: Valid request but no data (e.g., fund with no NAV history) → 200 with empty slice
  - **Network error**: Provider network failure → fallback if available, error if not
- **Instrumentation & Monitoring**:
  - Log each capability request: capability, type, params, primary provider, result (hit/miss/fallback)
  - Track latency per provider (first call vs cached)
  - Monitor cache size and evictions
  - Alert on repeated provider failures (may indicate upstream issues)
  - Track fallback frequency (should be rare; indicates config or provider issue)
