**Description**:
Implement a LangGraph-style orchestration runtime inside the Go wealth-management-engine so AI features move from one-off demos to a reusable graph/state system.

- **Features**:
  - `chat`: keeps conversation state across turns and supports tool execution inside the same session/thread.
  - `json`: returns validated JSON responses while still allowing the model to inspect selected skill definitions and call registered tools.
- **Skill Source**: Discover skills from `.agents/skills/<skill-name>/SKILL.md`.
- **Context Control**: Support whitelist and regex filters so only selected skills are loaded into the active LLM context.
- **Tool Backends**: Support heterogeneous tool executors, including native Go function tools and MCP-backed tools behind a common registry.
  **Acceptance Criteria**:
- Endpoint(s) expose a stable session/thread identifier so follow-up chat requests continue prior conversation state.
- Chat feature persists and reuses prior turns inside the same orchestration thread.
- JSON feature returns a valid parsed Go struct/JSON payload and can still trigger tool execution before final synthesis.
- Skill loader can enumerate `.agents/skills/*/SKILL.md` and filter by explicit whitelist and regex pattern.
- Tool registry can execute at least one native Go tool and one MCP-backed tool through the same orchestration interface.
- Orchestration tests cover message-state continuation, skill filtering, tool execution, and JSON response guarantees.

**Implementation Notes**:

- **Architecture**:
  - Keep Hexagonal boundaries strict:
    - `domain/` for thread/session, graph state, tool call/result, skill metadata.
    - `port/` for orchestration runtime, skill repository, tool registry, thread store.
    - `service/ai/...` for graph execution and session coordination.
    - `adapter/` for filesystem skill loading, function-tool adapters, MCP tool adapters, and Fiber handlers.
- **Graph Runtime Direction**:
  - Model a LangGraph-style loop with explicit state transitions:
    1. Load thread state
    2. Resolve active skills/tools
    3. Ask LLM for next action
    4. Execute tool call(s)
    5. Append results to thread
    6. Synthesize final chat or JSON response
- **Conversation Memory**:
  - Use append-only turn history keyed by `thread_id`/session ID.
  - Checkpoint state after each user/assistant/tool turn.
- **Skill Filtering**:
  - Support both explicit skill allow-lists and regex matching against skill names.
  - Default behavior must avoid loading every skill into prompt context.
- **Tool Registry**:
  - Unify local function tools and MCP tools under a shared execution contract.
  - Reuse existing MCP server concepts where practical rather than introducing a parallel protocol.
- **API Direction**:
  - Keep real API routes under `/api/ai/*`.
  - Preserve existing streaming/JSON capabilities while introducing graph-based chat/state orchestration.
- **Testing**:
  - Add service-level tests for graph flow and session continuation.
  - Add adapter-level tests for skill discovery/filtering and MCP/local tool dispatch.
  - Add Fiber handler tests for session-aware chat and JSON orchestration endpoints.
