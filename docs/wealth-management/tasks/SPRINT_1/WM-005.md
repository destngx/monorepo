**Description**:
Connect the Go backend to the existing Upstash Redis instance to maintain the < 300ms latency requirement.

- **Logic**: Use `go-redis` or equivalent.
  **Acceptance Criteria**:
- Go backend can `SET` and `GET` keys from the same Redis instance used by the legacy app.
- Cache invalidation logic ported to Go.

**Implementation Notes**:

- **Client Strategy**: Implemented an Upstash REST adapter in Go (equivalent to `go-redis` for this environment) to reuse the same legacy credentials:
  - `UPSTASH_REDIS_REST_URL`
  - `UPSTASH_REDIS_REST_TOKEN`
- **Hexagonal Wiring**:
  - `adapter/cache/upstash_client.go` implements cache operations (`set`, `get`, `keys`, `del`, `ping`).
  - `adapter/market/vnstock/provider.go` implements `vnstock` as one market provider implementation under a provider-agnostic market domain.
  - `service/cache_service.go` implements cache invalidation (`keys` + `delete` loop).
  - `service/market_provider_service.go` exposes provider-agnostic market provider health service.
  - `adapter/fiber/cache_handler.go` exposes cache endpoints:
    - `GET /api/cache/health`
    - `POST /api/cache/set`
    - `GET /api/cache/get/:key`
    - `DELETE /api/cache/invalidate?pattern=...`
  - `adapter/fiber/market_provider_handler.go` exposes provider endpoint:
    - `GET /api/external/market/providers/:provider/health`
    - Backward-compatible: `GET /api/external/vnstock/health`
- **TDD/BDD Tests**:
  - BDD-style adapter tests for `SET` and `GET` request behavior in `adapter/cache/upstash_client_test.go`.
  - BDD-style service tests for invalidation behavior in `service/cache_service_test.go`.
- **Nx Command**:
  - Updated `test` target in `apps/wealth-management-engine/project.json`:
    - `bunx nx run wealth-management-engine:test`
  - Uses repo-local Go caches under `tmp/` for sandbox-compatible execution.
